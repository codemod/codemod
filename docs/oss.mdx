Codemod provides open source toolkits that help framework and library maintainers create, publish, and maintain official codemods for their users.

With Codemod, maintainers can confidently ship new releases without worrying about long-term backward compatibility or keeping old versions alive. Codemods give users a safe, automated path to upgrade and stay current with the latest APIs and features.

Codemod is already used by popular projects like React, Node.js, ESLint, Express, Nuxt, and others to power migrations in their official upgrade guides.

## How it works

1. **[Write codemods](/workflows/quickstart)** that automate breaking changes using jssg, YAML rules, or multi-step workflows
2. **[Publish to the registry](/publishing)** under your framework's namespace using the CLI or CI/CD
3. **Users run them** with a single [CLI](/cli) command or through [Codemod platform](/platform/campaigns)




<Info>
  Want to add your framework? [Ping us on Slack](https://go.codemod.com/community) or start building below.
</Info>

## Getting started

Codemods can be built using:
- **[CLI](/cli)** - Scaffold and build locally with `npx codemod init`
- **[MCP](/model-context-protocol)** - AI-assisted creation in your IDE
- **[Studio](/platform/codemod-studio)** - Web-based prototyping with Codemod AI

This guide uses the CLI. The workflow is the same regardless of how you create the codemod.

### Build your first codemod

<Frame>
  <MigrationsTemplateDemo />

<Steps>
  <Step title="Scaffold a new package">
    ```bash
    npx codemod init
    ```

    The CLI walks you through setup:
    - **Codemod type** - jssg (simple transforms) or multi-step workflow (complex migrations)
    - **Target language** - TypeScript, JavaScript, Python, etc.
    - **Package name** - Use `@your-framework/codemod-name` for scoped packages

    This creates a folder with the required structure:

    ```plaintext
    my-codemod/
    ├── codemod.yaml        # Package metadata
    ├── workflow.yaml       # Workflow definition
    └── scripts/
        └── codemod.ts      # Your transformation logic
    ```
  </Step>

  <Step title="Write your transformation">
    Open `scripts/codemod.ts` and write your transformation. Here's a minimal example that replaces `console.log` with `logger.log`:

    ```ts scripts/codemod.ts
    import type { Transform } from "codemod:ast-grep";
    import type TSX from "codemod:ast-grep/langs/tsx";

    const transform: Transform<TSX> = (root) => {
      const node = root.root();
      const matches = node.findAll({ pattern: "console.log($$$ARGS)" });

      if (matches.length === 0) return null;

      const edits = matches.map((match) => {
        const args = match.getMultipleMatches("ARGS").map(a => a.text()).join(", ");
        return match.replace(`logger.log(${args})`);
      });

      return node.commitEdits(edits);
    };

    export default transform;
    ```

    See the [jssg quickstart](/jssg/quickstart) for patterns and API details.
  </Step>

  <Step title="Test locally">
    Run your codemod against a target directory:

    ```bash
    npx codemod workflow run -w ./my-codemod/ -t ./path/to/test-repo 
    ```

    For automated testing, create input/expected fixtures:

    ```plaintext
    tests/
      basic-case/
        input.ts       # Before
        expected.ts    # After
    ```

    Then run:

    ```bash
    npx codemod jssg test ./scripts/codemod.ts -l tsx
    ```

    See [testing documentation](/jssg/testing) for more options.
  </Step>
</Steps>

### Publish to the registry

Once your codemod works locally, publish it so your users can run it.

<Steps>
  <Step title="Login">
    ```bash
    npx codemod login
    ```

    This opens a browser for authentication. You can sign in with GitHub, GitLab, or SSO.
  </Step>

  <Step title="Publish">
    ```bash
    npx codemod publish
    ```

    Your codemod is now live in the [Codemod Registry](https://go.codemod.com/registry). Users can run it with:

    ```bash
    npx codemod @your-framework/codemod-name
    ```
  </Step>
</Steps>

<Tip>
  Use [scoped package names](/platform/registry#package-naming) (e.g., `@nuxt/v3-to-v4`) to group codemods under your framework's namespace.
</Tip>

### CI/CD and automation

For ongoing codemod development, use the [migrations template](https://github.com/codemod/migrations-template) to get a ready-to-use repository with GitHub Actions for validation and publishing.

<Steps>
  <Step title="Create your codemods repository">
    Fork or use the [migrations template](https://github.com/codemod/migrations-template) in your GitHub organization.
  </Step>

  <Step title="Link your organization">
    1. [Sign in](https://app.codemod.com) to Codemod
    2. [Create an organization](https://app.codemod.com/global/organizations/create) matching your GitHub org name
    3. Install the [Codemod GitHub App](https://github.com/apps/codemod)
    
    This enables [trusted publishing](/publishing#trusted-publishers)—passwordless publishing from GitHub Actions with no API keys to manage.
  </Step>

  <Step title="Configure your workflow">
    For trusted publishing, ensure your workflow has OIDC permissions:
    
    ```yaml
    permissions:
      id-token: write
      contents: read

    steps:
      - uses: actions/checkout@v4
      - uses: codemod/publish-action@v1
    ```
    
    <Tip>
      Prefer API keys instead? [Create one](https://app.codemod.com/api-keys) and add it as a repository secret named `CODEMOD_API_KEY`. See [publishing](/publishing) for all options.
    </Tip>
  </Step>

  <Step title="Push and publish">
    Add codemods to your repository. The GitHub Action validates on PR and publishes on release.
  </Step>
</Steps>
