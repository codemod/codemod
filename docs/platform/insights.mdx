Insights turns code data into dashboards. Use it to answer questions like:

- How many deprecated imports are left?
- Which repositories still need a migration?
- Which packages and versions are installed?
- Is adoption improving over time?

## Quick start

<Steps>
  <Step title="Create a dashboard">
    Go to **Insights**, then create a dashboard for one migration, code health area, or team.
  </Step>
  <Step title="Add a widget">
    Click **Add Widget** and choose how you want to display the result.

    <Frame>
      <img
        src="/images/insights/widget-types-light.png"
        alt="Add widget menu with timeseries, single number, table, and notes"
        className="block dark:hidden"
      />
      <img
        src="/images/insights/widget-types-dark.png"
        alt="Add widget menu with timeseries, single number, table, and notes"
        className="hidden dark:block"
      />
    </Frame>
  </Step>
  <Step title="Choose repositories">
    Select one or more repositories. For reusable dashboards, use a repository template variable.
  </Step>
  <Step title="Choose a data source">
    Pick JSSG, ast-grep, or dependencies. Configure the query, preview the result, then save the widget.
  </Step>
</Steps>

## Widget types

Use the widget type that matches the question:

| Widget | Use it for |
| --- | --- |
| **Timeseries chart** | Progress over time, such as deprecated API usage trending down. |
| **Single number** | One current value, such as files left to migrate. |
| **Table view** | Breakdowns by repository, directory, package, owner, or metric key. |
| **Notes** | Context, next actions, caveats, and links. |

Every query starts with a data source.

| Source | Use it when |
| --- | --- |
| **JSSG Codemod** | You need custom code analysis, semantic checks, or metrics emitted from a codemod. |
| **ast-grep (YAML)** | You need a structural syntax match without writing TypeScript. |
| **Dependencies** | You need package, version, ecosystem, workspace, or dependency depth data. |

### JSSG metrics

JSSG queries can count files or read named metrics emitted by the codemod. A metric can also include cardinality keys, for example `name`, `component`, `api`, or `package`.

Use metric cardinality when you want a table such as "deprecated prop usage by component name".

### ast-grep rules

ast-grep queries count matches from a YAML rule. They are good for direct syntax patterns, such as a specific import, JSX prop, or function call.

### Dependencies

Dependency queries use indexed package data from the selected repositories.

Grouping controls the shape of the result.

  <img

Common groups:

| Group by | What it shows |
| --- | --- |
| **Repository** | Which repos have the signal. |
| **Directory** | Where the work is located in a repo. |
| **Codeowner** | Which team likely owns the work. |
| **Metric cardinality key** | A custom dimension emitted by JSSG, such as component name or API name. |

Use a table widget when you want grouped results that people can scan and sort.

Use glob filters to remove tests, generated files, fixtures, or directories that should not count toward migration progress.

## Show progress with a formula

Single number widgets can combine multiple query lines with a formula.

<Frame>
  <img
    src="/images/insights/progress-formula-light.png"
    alt="Single number widget with two JSSG queries and a progress formula"
    className="block dark:hidden"
  />
  <img
    src="/images/insights/progress-formula-dark.png"
    alt="Single number widget with two JSSG queries and a progress formula"
    className="hidden dark:block"
  />
</Frame>

A common pattern:

1. Query `a`: count the old pattern at an earlier snapshot, such as one month ago.
2. Query `b`: count the old pattern at the latest commit.
3. Formula: `1 - (b / a)`.
4. Format the result as a percent.

This shows progress toward removing the old pattern. If the old count was `100` and the latest count is `25`, the formula returns `75%`.
