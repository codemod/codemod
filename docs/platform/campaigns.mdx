Campaigns run codemods across one or more repositories. Codemod creates tasks, opens pull requests, tracks CI checks, and gives you one place to follow the rollout.

Use a campaign when you want to move from "we found the work" to "we are applying the change".

## Create a campaign

<Steps>
  <Step title="Choose a codemod">
    Go to **Campaigns**, click **Create Campaign**, and choose a codemod from the registry.

    Use a package like `debarrel` for import cleanup, or choose another codemod package that matches your migration.
  </Step>
  <Step title="Select target repositories">
    Pick the GitHub or GitLab repositories you want to update. You can run against one repository first, then expand to more repositories after the first run looks good.

    <Frame>
      <img src="/images/campaigns/select-repositories.png" alt="Repository picker for a campaign" />
    </Frame>
  </Step>
  <Step title="Configure options">
    Fill in any parameters the codemod exposes. Some campaigns also include rollout options such as AI review, sharding, or CI recovery.
  </Step>
  <Step title="Review and start">
    Review the selected package, repositories, campaign name, parameters, and recovery settings. Then create the campaign.

    <Frame>
      <img src="/images/campaigns/review-campaign.png" alt="Campaign review page before creating the campaign" />
    </Frame>
  </Step>
</Steps>

## Follow the rollout

The campaign detail page shows the workflow, generated tasks, pull requests, and CI status.

<Frame>
  <img src="/images/campaigns/campaign-detail.png" alt="Campaign detail page with workflow tasks, generated pull request, and passing checks" />
</Frame>

Use this page to answer:

- Which workflow step is running?
- Which task created a pull request?
- Did CI pass on the generated PR?
- Does a task need to be rerun or reviewed?

## How campaigns use workflows

Campaigns are powered by [Codemod workflows](/workflows/introduction). A workflow defines the stages of a migration.

| Concept | Meaning |
| --- | --- |
| **Node** | A stage in the workflow, such as analyzing a repository or applying a transform. |
| **Step** | One action inside a node, such as running a JSSG codemod or a shell command. |
| **Task** | A runtime instance of a node. Matrix workflows can create multiple tasks. |
| **Pull request** | The branch and PR created when a task produces code changes. |

Most users do not need to edit workflow files to run a campaign. Choose a registry codemod, select repositories, and review the generated plan.

## Roll out safely

Start small, then expand:

1. Run the codemod against one repository.
2. Review the generated pull request.
3. Check CI status on the campaign detail page.
4. Run the campaign against more repositories.

For large migrations, use sharding to split work into smaller pull requests. You can group shards by directory or code owner so each team gets a reviewable slice.

## CI checks and recovery

Generated pull requests show supported CI checks inline. Passing checks appear on the task card so you can see whether the generated PR is ready to review.

If a check fails and CI recovery is enabled, Codemod can try to repair the generated PR or report what failed. You can inspect logs from the campaign page before deciding what to do next.
